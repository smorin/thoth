## 1. Goal

Ship the automation-and-scripting half of v3.0.0:

1. Add `thoth completion {bash,zsh,fish}` subcommand with `--install`, `--manual`, `--force`, `--json` flags. Backed by dynamic completer functions in `completion/sources.py`.
2. Add `--json` to every data/action admin command via the B-deferred per-handler `get_*_data() -> dict` extraction pattern, with envelope contract centralized in `json_output.py`.
3. Wire `shell_complete=` callbacks on registered subcommand options for full TAB-completion coverage of operation IDs, mode names, config keys, provider names, and (P18 forward-compat) mode kinds.

PR3 is purely additive. Once both PR2 and PR3 land on `main`, release-please tags v3.0.0.

## 2. Motivation

Per the original P16 design (decisions Q4 completion, Q5 `--json`) and the user's restated direction during PR3 brainstorming:

- **Shell completion is the most-requested CLI ergonomic.** Tab-completing operation IDs from `~/.thoth/operations/` is the killer scripted-use win that PR1+PR2 set up but didn't deliver.
- **Universal `--json` enables clean script composition.** Currently `config list` and `modes list` have `--json`; `status`, `list`, `providers`, `init`, `ask`, `resume` do not. The inconsistency forces script writers to parse Rich-rendered text. Closing the gap completes the spec's automation motivator.
- **Mode-aware non-blocking JSON (Option E)** turns `--json` into a uniform "snapshot" contract across all commands. Long-running operations under `--json` submit-and-exit; pollable via `resume --json`. Eliminates the streaming-vs-buffered tension and makes `--json` runs safe to chain in pipelines.

## 3. Out of scope

- **New subcommands beyond `completion`** — PR2 already shipped `ask`/`resume`. No further verbs in PR3.
- **Removing anything** — PR3 is additive only (matches original spec §5.3 "PR3 — Nothing removed; pure addition").
- **Reworking exit codes** — still 0/1/2/6/7. Granular `error.code` strings live inside JSON envelopes only (per spec §8.3).
- **Wrapping completion scripts or help text in success JSON** — `completion <shell>` (no install) keeps its raw shell-script stdout for `eval "$(...)"` use. `help` stays human-only with no `--json` flag.
- **Migrating `interactive.py::SlashCommandCompleter` to import from `completion/sources.py`** — listed as polish opportunity, not blocker. Future PR.
- **Per-shell completion convention detection** (Q3-C from brainstorming, rejected in favor of A) — `--install` writes to rc file with fenced markers uniformly across shells.
- **PowerShell completion support** — only bash/zsh/fish are required. PowerShell can be added in a follow-up if requested.
- **`completion --uninstall` subcommand** — uninstall is documented as a `sed` one-liner (the fenced markers make this trivial). A real `--uninstall` flag is a follow-up.

## 5. Architecture

### 5.1 The shape

PR3 ships **two new surfaces** + **per-handler refactor work**:

1. **`completion/` package** (new directory) — generates shell init scripts for bash/zsh/fish + provides dynamic completer data sources.
2. **`json_output.py` module** (new file) — single source of truth for the JSON envelope contract.
3. **B-deferred per-handler `get_*_data()` extraction** — for each handler that gains `--json`, extract a pure data function alongside the Rich-printing function. Subcommand wrapper picks based on `--json`. The `as_json` flag NEVER appears in handler signatures.

Dispatch model from PR1+PR2 is unchanged. `@click.group(cls=ThothGroup)` stays. Subcommand registrations grow by one (`completion`).

## 6. Components

### 6.1 `json_output.py` (NEW)

**Purpose:** Single source of truth for JSON envelope contract per spec §6.4.

**Interface:**
```python
def emit_json(data: dict, *, exit_code: int = 0) -> NoReturn:
    """Emit success envelope: {"status":"ok","data":{...}} and sys.exit."""

def emit_error(
    code: str,
    message: str,
    details: dict | None = None,
    *,
    exit_code: int = 1,
) -> NoReturn:
    """Emit error envelope: {"status":"error","error":{...}} and sys.exit."""
```

**Depends on:** `json` (stdlib), `sys` (stdlib). Framework-free. ~40 LOC.

### 6.2 `completion/script.py` (NEW)

**Purpose:** Generate shell init scripts. Wraps Click's `_THOTH_COMPLETE=<shell>_source thoth` machinery.

**Interface:**
```python
def generate_script(shell: Literal["bash", "zsh", "fish"]) -> str
def fenced_block(shell: str) -> str
```

**Depends on:** Click 8.x completion machinery. ~80 LOC.

### 6.3 `completion/install.py` (NEW)

**Purpose:** Implement Q3-A install behavior — write to rc file with fenced markers, detect existing block, prompt-before-overwrite, support manual mode.

**Interface:**
```python
@dataclass
class InstallResult:
    action: Literal["written", "preview", "skipped"]
    path: Path
    message: str

def install(
    shell: Literal["bash", "zsh", "fish"],
    *,
    force: bool = False,
    manual: bool = False,
    rc_path: Path | None = None,
) -> InstallResult
```

**Behavior matrix** (covered by Category C tests):
| `--manual` | `--force` | TTY | Action |
|---|---|---|---|
| F | F | T | Detect → preview → prompt → write |
| F | F | F | Refuse with INSTALL_REQUIRES_TTY |
| F | T | * | Write/overwrite silently |
| T | F | * | Print block + instructions; never write |
| T | T | * | `BadParameter("--manual mutex with --force")` |

**Depends on:** `completion/script.py`, `pathlib`, `os.path.expanduser`, `sys.stdin.isatty()`. ~120 LOC.

### 6.4 `completion/sources.py` (NEW)

**Purpose:** Pure data functions for shell-completion callbacks. Each takes Click's `(ctx, param, incomplete)` signature.

**Interface:**
```python
def operation_ids(ctx, param, incomplete: str) -> list[str]
def mode_names(ctx, param, incomplete: str) -> list[str]
def config_keys(ctx, param, incomplete: str) -> list[str]
def provider_names(ctx, param, incomplete: str) -> list[str]
def mode_kind(ctx, param, incomplete: str) -> list[str]  # P18 forward-compat — currently dead code
```

**Depends on:** `BUILTIN_MODES`, `CheckpointManager.list_operation_ids()`, `paths.user_config_file()`. ~80 LOC.

### 6.5 `cli_subcommands/completion.py` (NEW)

**Purpose:** Click subcommand wiring `script.py` and `install.py`.

**Interface:**
```python
@click.command(name="completion")
@click.argument("shell")  # validated in body, NOT click.Choice
@click.option("--install", is_flag=True)
@click.option("--force", is_flag=True)
@click.option("--manual", is_flag=True)
@click.option("--json", "as_json", is_flag=True)
def completion(shell, install, force, manual, as_json): ...
```

`shell` is intentionally NOT `click.Choice` — invalid shell needs to be emit-able as `UNSUPPORTED_SHELL` JSON envelope, which Click's choice validation can't do. Body validates against `{"bash", "zsh", "fish"}` and emits the right error shape.

**Depends on:** `completion/script.py`, `completion/install.py`, `json_output.py`. ~50 LOC.

### 6.6 Per-handler `get_*_data()` extractions

For each handler that gains `--json`, extract a pure data function:

**Pattern:**
```python
def get_status_data(operation_id: str, *, ctx_obj=None) -> dict | None:
    """Pure data function. Returns dict for JSON envelope, or None if not found."""
    op = checkpoint_load(operation_id)
    if op is None:
        return None
    return {"operation_id": op.id, "status": op.state, ...}

def show_status(operation_id: str, *, ctx_obj=None) -> None:
    """Existing Rich-printing function; refactored to call data function then format."""
    data = get_status_data(operation_id, ctx_obj=ctx_obj)
    if data is None:
        console.print("[red]Error:[/red] Operation not found")
        return
    console.print(f"[bold]Status:[/bold] {data['status']}")
    # ... rich rendering
```

**Critical invariant** (CI lint enforces): `as_json` never appears in any handler signature. The flag lives at the subcommand wrapper layer only.

### 6.7 `cli_subcommands/ask.py` (Option E branching)

```python
def ask(ctx, prompt_args, ..., as_json, mode_opt, async_mode):
    # ... existing prompt resolution + mutex checks ...
    if as_json:
        if _is_background_mode(mode) or async_mode:
            op_id = _submit_research(...)  # submit, no polling
            emit_json({"operation_id": op_id, "status": "submitted", "mode": mode, "provider": provider})
        else:
            data = _run_research_get_data(...)  # synchronous; full result
            emit_json(data)
    else:
        # Existing path — streaming Rich output
        ...
```

`_is_background_mode(mode)` reads from `BUILTIN_MODES[mode]["kind"]` (P18 forward-compat) OR falls back to existing `is_background_model(model)` substring check.

### 6.8 `cli_subcommands/resume.py` (Option E snapshot)

```python
def resume(ctx, operation_id, ..., as_json):
    if as_json:
        data = thoth.run.get_resume_snapshot_data(operation_id)
        if data is None:
            emit_error("OPERATION_NOT_FOUND", ..., exit_code=6)
        if data["status"] == "failed_permanent":
            emit_error("OPERATION_FAILED_PERMANENTLY", data["last_error"], details=data, exit_code=7)
        emit_json(data)  # all other states: ok envelope with data.status reflecting actual state
    else:
        # Existing blocking-retry behavior
        _run_maybe_async(thoth_run.resume_operation(...))
```

`get_resume_snapshot_data()` is the new pure data function — reads checkpoint, returns dict, NEVER advances state.

## 7. Data flow

See **Section 3** of brainstorming for the 8 paths in detail. Summary:

1. **`thoth completion <shell>`** (no flags) — `script.py::generate_script()` → stdout
2. **`thoth completion <shell> --install`** (TTY) — `install.py::install()` → detect/prompt/write rc file
3. **`thoth completion <shell> --install --manual`** — print fenced block + instructions; never write
4. **`thoth completion <shell> --install --json`** — wrap result in JSON envelope
5. **`<command> --json`** for synchronous admin commands — handler's `get_*_data()` → `emit_json(data)` or `emit_error(...)`
6. **`thoth ask --json` immediate mode** — sync run, full envelope with result inline
7. **`thoth ask --json` background mode** — auto-async; submit + return op-id envelope (no polling)
8. **`thoth resume --json`** — pure snapshot; never advances state; envelope reflects current checkpoint state

The `as_json` flag NEVER reaches handlers (per spec §7.2 critical invariant). Handlers below the subcommand wrapper see only data, never JSON-vs-Rich choice.

## 8. Error handling

### 8.1 JSON envelope contract — error codes catalog

| Code | Used by | Exit | Meaning |
|---|---|---|---|
| `OPERATION_NOT_FOUND` | `status`, `resume` | 6 | Op ID not in checkpoint store |
| `OPERATION_FAILED_PERMANENTLY` | `resume` | 7 | Permanent failure state |
| `JSON_REQUIRES_NONINTERACTIVE` | `init` | 2 | `init --json` without `--non-interactive` |
| `EDITOR_FAILED` | `config edit` | 1 | `$EDITOR` exited non-zero |
| `UNSUPPORTED_SHELL` | `completion` | 2 | Shell name not in {bash, zsh, fish} |
| `INSTALL_REQUIRES_TTY` | `completion --install` | 2 | non-TTY + no `--force` + no `--manual` |
| `INSTALL_FILE_PERMISSION` | `completion --install` | 1 | Can't write to rc file |
| `KEY_NOT_FOUND` | `config get` | 1 | Config key doesn't exist |
| `INVALID_LAYER` | `config get --layer` | 2 | Click choice validation (already wired in PR2) |
| `PROVIDER_FAILURE` | `ask`, `resume` | 1 | Upstream provider error |
| `API_KEY_MISSING` | `ask`, `resume`, `providers check` | 1 | Required key not set |
| `INTERRUPTED` | any | 130 | SIGINT during `--json` run |

### 8.2 Exit codes (unchanged from PR2)

| Code | Meaning |
|---|---|
| 0 | Success |
| 1 | Runtime error |
| 2 | Usage error (Click + custom mutex/gating) |
| 6 | Operation not found |
| 7 | Operation permanently failed |
| 130 | SIGINT (Unix convention: 128 + signum) |

No new exit codes vs PR2's catalog. JSON envelopes provide granular `error.code` strings for programmatic distinction.

### 8.3 `--json` invariants

The `tests/test_json_envelopes.py` parametrize-list provides one test per command. Each test asserts:

1. Output parses as JSON (`json.loads(r.output)` doesn't raise)
2. Top-level is a dict
3. Has `"status"` key with value `"ok"` or `"error"`
4. If `"ok"`: has `"data"` key (dict)
5. If `"error"`: has `"error"` key with `"code"` (str) and `"message"` (str); optionally `"details"` (dict)

The CI lint rule walks `cli_subcommands/` AST to ensure `JSON_COMMANDS` parametrize-list stays in sync with reality.

### 8.4 Non-blocking JSON guarantee (Option E)

`--json` is documented as non-blocking + snapshot-shaped. Tests in Category G assert timing properties:
- `ask --json --mode deep_research` returns within 5 seconds (submit-and-exit)
- `resume --json` on running op never enters polling loop
- `resume --json` on `recoverable_failure` returns `status:"ok"` (not error)

### 8.5 `recoverable_failure` envelope shape

`resume --json` on a `recoverable_failure` checkpoint:
```json
{
  "status": "ok",
  "data": {
    "operation_id": "abc123",
    "status": "recoverable_failure",
    "mode": "deep_research",
    "last_error": "rate limit exceeded",
    "retry_count": 2
  }
}
```

`status:"ok"` because the COMMAND succeeded. `data.status:"recoverable_failure"` describes the operation. To advance/retry, user runs `thoth resume OP_ID` WITHOUT `--json` (today's blocking-retry behavior).

### 8.6 SIGINT handling

`signals.py::handle_sigint` continues from PR2. `--json` runs catch `KeyboardInterrupt` at the subcommand wrapper, emit `emit_error("INTERRUPTED", ..., exit_code=130)`, exit cleanly.

### 9.3 TDD ordering within each commit

Per CLAUDE.md "Test-driven development":
1. Failing test first (red)
2. Minimal implementation (green)
3. Widen one step
4. `just check && uv run pytest tests/<file>.py` before commit

For per-handler `get_*_data()` extractions: extract test → red → write data function → green → refactor existing handler to call it → green again → commit.

### 9.4 What's NOT tested (deliberate)

- Click's own completion machinery (trust the framework)
- Subprocess invocation through actual shell (testing `eval "$(thoth completion zsh)"` requires spawning subshells — too fragile across CI environments)
- JSON output byte-for-byte stability across Click versions (only assert structural shape)
- Performance of `get_*_data()` functions (pure dict-builders; if slow, it's a downstream provider issue)
