## 1. Goal

Land v3.0.0 by:
1. Adding two new first-class Click subcommands: `thoth resume OP_ID` (replaces the `--resume` flag) and `thoth ask "prompt"` (canonical scripted research entry point).
2. Removing every flag-style shim catalogued by the comprehensive audit: `--resume`, the `providers --` separator + in-group `--list/--models/--keys/--check/--refresh-cache/--no-cache` shim, the `modes --json/--show-secrets/--full/--name/--source` PR1.5 hidden-subcommand shim, and the `thoth --help <topic>` parse-time hijack.
3. Applying a 13-item cleanup batch from the audit's untested-but-in-production findings — converting silent ambiguities, silent drops, and undocumented behaviors into explicit `BadParameter` rejections or Click-default exit codes.
4. Splitting the security-adjacent `--raw` flag's masking-bypass into two semantically clean flags: `--raw` (formatting only) and `--show-secrets` (override masking).

PR2 is the breakage half of v3.0.0; PR3 will follow with the additive automation polish.

## 2. Motivation

Per the original PR2 spec (decisions Q2/Q3 from `2026-04-25-promote-admin-commands-design.md`) and the user's restated direction during PR2 brainstorming:

- **First-class verbs for automation.** Scripts should `thoth resume OP_ID` and `thoth ask "..."` — short, predictable, discoverable. The flag-style `thoth --resume OP_ID` form was a pseudo-dispatch hack that hid the verb-noun structure.
- **No silent shims in v3.0.** The PR1.5 stabilization re-introduced shims to keep behavior bit-stable through the refactor. v3.0 is the right window to retire them all together — once. Living with shims means living with their inconsistencies (e.g., `providers --check` undocumented alias, `--refresh-cache + --no-cache` silent ambiguity, `modes list --name X --source Y` silent drop) forever.
- **No functionality loss in the migration.** Every removed form must either map to a documented new form OR be explicitly retired with documented justification. The 167-task PROJECTS.md ledger and 95-test target are the verifiable contract that nothing silently disappears.

## 3. Out of scope

- `--json` for every admin command — already partially shipped in PR1.5; full coverage is PR3's deliverable per the original spec §10
- The `completion` Click subcommand and dynamic completers — PR3
- Per-handler `get_*_data() -> dict` extraction — PR3 (B-deferred pattern from original spec §6.5)
- Adding new mode-editing operations (`modes set/add/unset`) — those land in P12, not PR2
- Mode-positional dispatch (`thoth deep_research "topic"`) — kept per Q2-D of the original brainstorming (re-confirmed during the comprehensive audit's KEEP? review)
- Bare-prompt fallback (`thoth "..."`) — kept per the same Q2-D decision
- P18 (immediate vs background path split) coordination — independent; merge order doesn't matter
- Redesigning the legacy-form gating message format beyond Click-native (Q6-B3 locks the format)

## 5. Architecture

### 5.1 The shape

PR2 makes three structural changes on top of PR1's `@click.group(cls=ThothGroup)` foundation:

1. **Add `ask` and `resume` as first-class Click subcommands** in `cli_subcommands/`. Both are thin wrappers delegating to existing handlers.
2. **Remove every flag-style shim** with explicit gating callbacks per Q6-C1. Each removed form `ctx.fail(...)`s with a Click-native error containing `(use 'thoth NEW_FORM')`.
3. **Apply the Q5-A cleanup batch** — ~13 small additions of `BadParameter` rejections, exit-code adjustments, and dropped undocumented aliases.

The dispatch model from PR1 stays intact:

```
thoth <args>...
  │
  ├─ args[0] is a registered subcommand (now including ask, resume)?
  │    → Click routes to the subcommand. Done.
  │
  ├─ args[0] is in BUILTIN_MODES?
  │    → ThothGroup.invoke routes to research execution path.
  │
  └─ Else (bare prompt, or empty)
       → ThothGroup.invoke treats as default-mode prompt.
```

Bare-prompt and mode-positional paths remain bit-identical because Q3-C duplicates research flags onto `ask` AND keeps them on the group. `thoth --mode X "..."` and `thoth ask --mode X "..."` and `thoth "..."` all keep working.

## 6. Components

### 6.1 `cli_subcommands/_options.py` (NEW)

**Purpose:** Single source of truth for the 15-flag research-options decorator stack. Applied identically to the top-level group AND to `ask`, satisfying Q3-PR2-C without duplication.

**Interface:**
```python
def _research_options(f):
    for opt in _RESEARCH_OPTION_DECORATORS:  # ordered list of @click.option(...) decorators
        f = opt(f)
    return f
```

**Depends on:** `click` only. Pure decorator file.

**Size:** ~30 LOC.

### 6.2 `cli_subcommands/ask.py` (NEW)

**Purpose:** Canonical scripted research entry point. Per Q7-PR2-B accepts `nargs=-1` positional joined with single spaces. Per Q3-PR2-C inherits the full research options.

**Interface:**
```python
@click.command(name="ask")
@click.argument("prompt_args", nargs=-1)
@click.pass_context
@_research_options
def ask(ctx, prompt_args, mode_opt, prompt_opt, prompt_file, ...):
    """Run a research operation with the given prompt."""
    prompt = " ".join(prompt_args) if prompt_args else None
    if prompt and (prompt_opt or prompt_file):
        raise click.BadParameter("Cannot use --prompt or --prompt-file with positional prompt")
    if prompt_opt and prompt_file:
        raise click.BadParameter("Cannot use --prompt-file with --prompt")
    effective = prompt or prompt_opt or _read_prompt_file(prompt_file)
    if not effective:
        raise click.BadParameter("Provide a prompt: positional, --prompt, or --prompt-file")
    return _run_research_default(mode=mode_opt or "default", prompt=effective, ...)
```

**Depends on:** `cli_subcommands/_options.py`, `thoth.run.run_research` (existing handler), `thoth.cli._run_research_default` (PR1's bare-prompt path target).

**Size:** ~50 LOC.

### 6.3 `cli_subcommands/resume.py` (NEW)

**Purpose:** Canonical resume entry point per Q1-PR2-C (Tight + Honor). Accepts focused honor-set; all other globals naturally rejected by Click.

**Interface:**
```python
@click.command(name="resume")
@click.argument("operation_id", metavar="OP_ID")
@click.option("--verbose", "-v", is_flag=True)
@click.option("--config", "-c", "config_path", help="...")
@click.option("--quiet", "-Q", is_flag=True)
@click.option("--no-metadata", is_flag=True)
@click.option("--timeout", "-T", type=float)
@click.option("--api-key-openai", help="...")
@click.option("--api-key-perplexity", help="...")
@click.option("--api-key-mock", help="...")
@click.pass_context
def resume(ctx, operation_id, verbose, config_path, quiet, no_metadata, timeout, api_key_openai, api_key_perplexity, api_key_mock):
    if not _checkpoint_exists(operation_id):
        click.echo(f"Error: Operation '{operation_id}' not found in checkpoint store.", err=True)
        ctx.exit(6)
    if config_path:
        thoth.config._apply_config_path(config_path)
    cli_api_keys = {"openai": api_key_openai, "perplexity": api_key_perplexity, "mock": api_key_mock}
    return thoth.run.resume_operation(operation_id, verbose=verbose, quiet=quiet, no_metadata=no_metadata, timeout_override=timeout, cli_api_keys=cli_api_keys, ctx=app_ctx)
```

**Depends on:** `thoth.run.resume_operation` (modified), `thoth.cli._build_app_context` (existing), `thoth.config._apply_config_path` (existing).

**Size:** ~40 LOC.

## 7. Data flow

PR2 introduces 5 new dispatch outcomes (paths 1, 2, 3, 4, 5 below) and modifies 2 (paths 6, 7). All other invocation forms are unchanged from PR1.5.

1. **`thoth ask "prompt"`** — Click → `ask` subcommand callback → join `nargs=-1` positional → mutex validation (positional vs `--prompt` vs `--prompt-file`) → `run_research(mode=mode_opt or "default", prompt=effective_prompt, ...)` with full research-options inheritance per Q3-PR2-C.
2. **`thoth resume OP_ID`** — Click → `resume` subcommand callback → `_checkpoint_exists` check (exit 6 if missing) → `_apply_config_path` if `--config` set → `resume_operation(operation_id, **honor_set)` per Q1-PR2-C.
3. **`thoth --resume OP_ID`** (REMOVED) — `ThothGroup.parse_args` early intercept (`--resume` / `-R` substring scan in args) → `ctx.fail("no such option: --resume (use 'thoth resume OP_ID')")` → exit 2 stderr.
4. **`thoth providers [-- ]--list/--models/--keys/--check/--refresh-cache/--no-cache`** (REMOVED) — `providers` subgroup callback → gating-loop scan over `ctx.args` → on match, `ctx.fail("no such option: {legacy_flag} (use '{new_form}')")` per Q6-C1.
5. **`thoth modes --json/--show-secrets/--full/--name/--source`** (REMOVED) — `modes` subgroup callback → identical gating-loop pattern routing each flag to `thoth modes list <flag>`.
6. **`thoth modes list --name X --source Y`** (MODIFIED) — both filters chain via `and` (intersection per Q5-A row 11.i); empty result returns exit 0, not error.
7. **`thoth config get KEY [--raw] [--show-secrets]`** (MODIFIED) — masking decision is now solely based on `--show-secrets` per Q4-PR2-D; `--raw` controls only output formatting (no JSON quoting, no Rich rendering).

**All other forms unchanged:** `thoth init`, `thoth status OP_ID`, `thoth list`, `thoth providers list/models/check`, `thoth config get/set/unset/list/path/edit`, `thoth modes list`, `thoth help [TOPIC]`, `thoth deep_research "topic"` (mode-positional), `thoth "prompt"` (bare-prompt), `thoth --version`, `thoth -i`, `thoth -m MODE -q PROMPT`.

The bare-prompt fallback continues to work because Q3-PR2-C keeps research flags on the group; `ThothGroup.invoke` reads from `ctx.params` (group-level flags) the same way it does today.

## 8. Error handling

### 8.1 Click-handled (free)

- Unknown subcommand → exit 2
- Bad option type → exit 2
- Missing required argument → exit 2 (`resume` requires OP_ID, `status` requires OP_ID)
- Reject-list options on `resume` → exit 2 (Click natively rejects undeclared options)
- Bare `thoth providers` / `thoth modes` no leaf → exit 2 (Q5-A rows 4, 5)
- `thoth status` no arg → exit 2 (Q5-A row 6; was 1 in PR1.5)

All error output goes to **stderr** per Q6-PR2-A1.

### 8.2 Custom gating (Q6-PR2-C1)

Each gating site uses inline `ctx.fail(f"no such option: {old} (use '{new_form}')")`. The `(use '...')` substring is the migration hint; gating tests assert on it.

| Removed form | Gating site | Migration message |
|---|---|---|
| `--resume`, `-R` | `ThothGroup.parse_args` early intercept | `(use 'thoth resume OP_ID')` |
| `providers -- --list` / `providers --list` | `providers` subgroup callback | `(use 'thoth providers list')` |
| `providers -- --models` / `providers --models` | same | `(use 'thoth providers models')` |
| `providers -- --keys` / `providers --keys` / `providers --check` | same | `(use 'thoth providers check')` |
| `providers -- --refresh-cache` / `providers --refresh-cache` | same | `(use 'thoth providers models --refresh-cache')` |
| `providers -- --no-cache` / `providers --no-cache` | same | `(use 'thoth providers models --no-cache')` |
| `modes --json` | `modes` subgroup callback | `(use 'thoth modes list --json')` |
| `modes --show-secrets` | same | `(use 'thoth modes list --show-secrets')` |
| `modes --full` | same | `(use 'thoth modes list --full')` |
| `modes --name X` | same | `(use 'thoth modes list --name X')` |
| `modes --source X` | same | `(use 'thoth modes list --source X')` |
| `--help <topic>` (parse-time hijack) | dropped; Click natively errors | `Got unexpected extra argument (X)` |

Per CLAUDE.md "Test-driven development":
1. Failing test first (red)
2. Minimal implementation to pass (green)
3. Widen to file-level pytest
4. `just check && uv run pytest tests/<file>.py` before commit

For migration commits (Category H), the migration test goes red first, the production code change makes it green, and the existing-test edits land in the same commit so the file stays self-consistent.
