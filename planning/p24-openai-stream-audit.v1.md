# P24 Task 6.1: OpenAI Responses API streaming-events audit

**Date**: 2026-05-03
**Owner**: P24 plan, Phase 6 Task 6.1
**Outcome**: SHIP (wire reasoning + citation events through `OpenAIProvider.stream()`)

## Question

Does the OpenAI Responses API stream emit reasoning-summary deltas and
annotation events during streaming, or only in the final response (read at
`get_result()` time)?

## Evidence

### SDK survey (openai-python 2.33.0)

Inspected `openai/types/responses/response_stream_event.py`. The
`ResponseStreamEvent` discriminated union enumerates 50+ event types,
including these directly relevant to the four-kind StreamEvent contract
(`text` / `reasoning` / `citation` / `done`):

- `ResponseTextDeltaEvent` — `type = "response.output_text.delta"`,
  carries `delta: str`. **Already handled** by current `stream()`.
- `ResponseReasoningSummaryTextDeltaEvent` — `type =
  "response.reasoning_summary_text.delta"`, carries `delta: str`,
  `summary_index: int`. Documented as "Emitted when a delta is added to
  a reasoning summary text." **NOT currently handled**.
- `ResponseOutputTextAnnotationAddedEvent` — `type =
  "response.output_text.annotation.added"`, carries `annotation: object`,
  `annotation_index: int`. Documented as "Emitted when an annotation is
  added to output text content." **NOT currently handled**.
- `ResponseCompletedEvent` — terminal; current `stream()` already emits
  its own `kind="done"` after the async-context-manager exits, which
  preserves the existing terminal-event guarantee.

(Also present: `ResponseReasoningTextDeltaEvent` with type
`"response.reasoning_text.delta"`, distinct from the *summary* deltas. We
prefer the summary stream — it matches what `get_result()` surfaces under
"## Reasoning Summary" today and what `submit()` enables via
`reasoning={"summary": "auto"}`.)

### Annotation payload shape

`AnnotationURLCitation` (in `openai/types/responses/response_output_text.py`)
defines URL citations with `type = "url_citation"`, `url: str`, `title: str`
(plus `start_index`/`end_index`). Other annotation variants —
`AnnotationFileCitation`, `AnnotationContainerFileCitation`,
`AnnotationFilePath` — lack a `url` field. The streaming
`annotation` object is typed as bare `object` (untyped union), so the
provider must defensively check for a `url` attribute / dict key, exactly
mirroring `get_result()`'s existing parser at `openai.py:602-610`.

### Citations of upstream behavior

The events are documented in the SDK's generated stub docstrings (the
SDK is generated by Stainless from the OpenAPI spec, so the type
declarations are authoritative for which events the API can emit).

## Decision: SHIP

Wire two new event types through `OpenAIProvider.stream()`:

1. `response.reasoning_summary_text.delta` → `StreamEvent(kind="reasoning",
   text=delta)`.
2. `response.output_text.annotation.added` → `StreamEvent(kind="citation",
   text=title, citation=Citation(title=title, url=url))`, **only** when
   the annotation carries a URL (i.e., `type == "url_citation"` or has
   `url` accessible via attr/key). Skip file/container/path annotations.

This brings OpenAI to four-kind parity with Perplexity (already
emits all four) and Gemini (Task 4.4, emits all four).

### Caveat: request-shape limitation (out of scope here)

`OpenAIProvider.stream()` currently builds `request_params` with only
`model`, `input`, and conditionally `temperature`. It does **not** set
`reasoning={"summary": "auto"}` (which `submit()` does) and does not
enable any tools (which `submit()` does for background/deep-research
models). This means that even with the new event-handler dispatch wired
in, the upstream API will not emit reasoning-summary or
annotation-added events for the typical immediate-path request unless
the request shape changes.

We deliberately do **not** modify the request shape under this audit
task — that would change runtime cost / latency / behavior. The
event-handler wiring is symbolic-parity correctness (cross-provider
stream contract) plus future-proofing against config changes (e.g.,
P25 / P26 may add reasoning-on-immediate or web-search-on-immediate
modes). A follow-up task should decide whether and how to enable
`reasoning={"summary": "auto"}` and `web_search_preview` on the
immediate-streaming path; that is a product decision, not an audit
finding.

The inline `stream()` docstring records this caveat so future
maintainers understand why the new event handlers might appear unused
during routine immediate runs.

### Test plan

A unit test in `tests/test_openai_errors.py` mocks
`client.responses.stream` as an async context manager whose iterator
yields a fixed sequence:

1. `response.output_text.delta` (delta=`"Hello "`)
2. `response.reasoning_summary_text.delta` (delta=`"Thinking step 1."`)
3. `response.output_text.delta` (delta=`"world."`)
4. `response.output_text.annotation.added` with a url_citation-shaped
   annotation
5. (terminal — context manager exits)

Asserts the provider emits, in order:

- `StreamEvent(kind="text", text="Hello ")`
- `StreamEvent(kind="reasoning", text="Thinking step 1.")`
- `StreamEvent(kind="text", text="world.")`
- `StreamEvent(kind="citation", citation=Citation(title=..., url=...))`
- `StreamEvent(kind="done", text="")`

This test pins the SHIP outcome; if the SDK ever changes the event
type-string spelling, the test fails fast.
